# Design History File (DHF) Template

> **Template Notice:** This document is an organization-level template. It is not a completed Design History File record until it has been copied into a product-specific controlled location, completed with accurate evidence, reviewed, and approved.

This folder is a **reusable, organization-wide Design History File template** maintained in the `iCRcompany/.github` repository. It provides a standardized structure for documenting design and development activities for regulated software products across the organization.

---

## What Is a Design History File?

A Design History File (DHF) — also referred to as a **Design and Development File** — is a collection of records that describes the design history of a finished device or software product. The DHF demonstrates that the design was developed in accordance with the approved design plan and applicable regulatory requirements (e.g., 21 CFR Part 820, ISO 13485, IEC 62304).

This template may produce a completed file called any of the following:

- **Design History File (DHF)**
- **DHF**
- **Design and Development File**

---

## Important: This Is a Template

This folder is a **reusable template** and is **not itself a completed product DHF**. It contains placeholder text and empty evidence tables. No historical approvals, test results, review outcomes, or regulatory evidence have been entered.

Product-specific information, evidence references, approvals, and completed records must be added by qualified personnel after copying this template into the applicable product repository.

---

## How to Use This Template

Each software product repository should receive its own copy of this template under:

```
docs/dhf/
```

Product-specific DHF records must be maintained in the applicable product repository or another approved controlled location.

---

## Recommended Use

1. Copy `dhf-template` into the product repository as `docs/dhf`.
2. Replace all placeholders with accurate, product-specific information.
3. Identify the software version or version range covered by this DHF.
4. Add references to existing design-control evidence (issues, pull requests, commits, tags, test records, risk records, quality documents, approval records).
5. Identify missing records as gaps in the **Identified Gaps or Required Actions** section of each document.
6. Create corrective or retrospective actions where required under the applicable quality procedure.
7. Review the completed DHF through a pull request and obtain required approvals.
8. Export the approved DHF to PDF and store in a controlled document location when needed.
9. Maintain the DHF as the software product changes.

---

## Supporting Evidence

GitHub issues, pull requests, commits, tags, releases, test records, risk records, quality documents, and approval records may be referenced as supporting evidence throughout this DHF.

**Hyperlinks alone are not sufficient.** Each reference should include stable identifying information such as:

- Repository name
- Issue number
- Pull request number
- Release tag
- Commit SHA
- Document number and revision
- Approval date
- Record status
- Controlled storage location

---

## AI-Generated Content

> AI-generated content (including content generated by GitHub Copilot or similar tools) must be reviewed for accuracy and approved by qualified personnel before being included in a completed DHF.

Copilot must **never** be used to fabricate:

- Historical approvals
- Test execution records
- Review attendance records
- Risk acceptance decisions
- Release approval records
- Validation results
- Any records that did not actually occur

---

## Gaps and Missing Evidence

Missing historical evidence must be identified as a gap in the relevant DHF section. Do not invent or reconstruct records that do not exist.

Retrospective assessments must be clearly labeled as retrospective and approved under the applicable quality procedure.

---

## Review and Approval

Completed DHF content should be reviewed through a pull request. Approved releases may be exported to PDF and stored in a controlled document location.

---

## Shared Records

Records that apply to multiple products may be referenced rather than duplicated. When referencing a shared record, include the full identifying information (repository, document number, revision, approval date, storage location) to ensure traceability.

---

## Product-Specific Customization

Product-specific customization is expected after copying this template. Add, remove, or adapt sections to match the product's regulatory classification, software safety class, applicable standards, and development history.

---

## Template Maintenance

This template is maintained in the `iCRcompany/.github` repository. Updates to the template do not automatically update product-specific DHF copies. Product teams should review the template periodically and incorporate relevant improvements through their change-control process.
